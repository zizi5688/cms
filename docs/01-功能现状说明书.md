## 一、系统概述

Super CMS 是一个**小红书矩阵号内容管理系统**，核心能力：

1. **多账号管理** — 每个小红书账号使用独立的 Electron `partition` 隔离 Cookie/Session
2. **批量内容创建** — 通过 CSV 导入 + 图片/视频素材，批量生成发布任务
3. **日历排期** — 将任务拖拽到日历上，设定发布时间
4. **自动发布** — 到达排期时间后，自动打开小红书创作者后台，通过 DOM 自动化完成上传和发布
5. **AI 裂变 (Remix)** — 对已有任务的图片做微调（裁切/亮度抖动）、标题/正文做同义词替换，生成"新"任务
6. **图片工具箱 (ImageLab)** — 九宫格切图、Real-ESRGAN 超分辨率、AI 水印去除

### 2.2 Draft vs Immediate 判定逻辑

**判定位置**: `publisher.ts` 的 `normalizeTask()` 函数 (行 56-79)

```
默认值 = 'draft' (行 77)
只有当 taskData.mode === 'immediate' 时才为 immediate
```

**在自动化脚本中的流转** (`xhs-automation.ts`):
- `mode === 'draft'` → 脚本会寻找并点击"存草稿"按钮
- `mode === 'immediate'` → 脚本会寻找并点击"发布"/"发布笔记"/"立即发布"按钮

**特殊路径 — publishDrafts**:
`cms.queue.publishDrafts` (publisher.ts:603) 专门用于"发布已存在的草稿"。它会判断：
- 如果任务 `status === 'draft_saved'` 或 `status === 'publish_failed'`，调用 `runImmediate()` → 实际调用 `runSchedule(task, false)`，使用 `type: 'publish_draft'` 指令
- 否则走正常 `publishTask()` 流程

**关键细节**: `dryRun` 参数默认为 `true`（行 76），这意味着通过 `publisher.publish` IPC 手动触发时，默认不会真正发布。只有队列自动执行和 publishDrafts 显式传 `false`。

### 2.3 定时心跳调度

**位置**: `index.ts:330-357`

```typescript
scheduleHeartbeatTimer = setInterval(() => {
  // 每 5 秒检查一次
  const dueTasks = taskManager.listDueTasks(now)
  // 按 accountId 分组
  for (const accountId of accountIds) {
    if (runningScheduleByAccount.has(accountId)) continue // 去重：同一账号不并发
    runQueue({ accountId, publisherService, taskManager })
  }

```

**去重机制**: `runningScheduleByAccount` 是一个 `Map<string, Promise<void>>`，确保每个账号同时只有一个队列在跑。Promise 完成后自动删除 key。

## 三、数据迁移机制

### 3.1 JSON → SQLite 迁移策略

**位置**: `sqliteService.ts:150-164`

系统启动时的迁移逻辑（`init()` 方法末尾）：

```
1. 检测 db.json 是否存在 且 db.json.bak 不存在
   → 是: 调用 migrateFromJSON()
        → 读取 JSON → INSERT OR IGNORE 到 SQLite
        → 将 db.json 重命名为 db.json.bak
        → 再复制一份回 db.json (保留原文件!)
           ⚠️ 这里逻辑是：rename → copyFile(bak, json)
           意味着迁移后 db.json 仍然存在，只是多了一个 .bak 标记

2. 检测 db.json.bak 是否存在 且 .dbjson_bak_merged 标记文件不存在
   → 是: 从 .bak 再合并一次 (防止 bak 中有更新的数据)
        → 写入 .dbjson_bak_merged 标记文件阻止重复合并
```

**为什么有两次合并？**
这是为了处理一种边界情况：在老版本中 db.json 可能被先备份为 .bak，但新数据又写回了 db.json。两次合并确保不丢数据。

### 3.2 迁移的字段映射

**位置**: `sqliteService.ts:187-440` (`migrateFromJSONFile`)

关键映射：
- JSON 的 `xhs_tasks` → SQLite 的 `tasks` 表
- JSON 的 `accounts` → SQLite 的 `accounts` 表
- JSON 的 `xhs_products` → SQLite 的 `products` 表
- `scheduledAt` 兼容多个字段名：`scheduledAt` / `scheduleTime` / `startTime`
- 任务 ID 冲突处理：如果 ID 相同但 `accountId` 不同，生成新 UUID
- 使用 `INSERT OR IGNORE` + 后续 `patchTask` UPDATE：已存在的记录只更新 `scheduledAt`、`images`、`videoPath`

## 四、关键功能模块实现

### 4.1 任务调度 (Queue System)

#### 4.1.1 QueueService 单例

**位置**: `services/queueService.ts`

状态流转：
```
pending ──► processing ──► published (成功+immediate)
   │            │           draft_saved (成功+draft)
   │            │
   │            └──► pending (失败且 retry_count < 3)
   │                 failed  (失败且 retry_count >= 3)
   │
   └── scheduled (有 scheduledAt 但未到时间)
```

#### 4.1.2 原子任务获取 (acquireNextTask)

**位置**: `queueService.ts:92-175`

```sql
UPDATE tasks
SET status = 'processing',
    locked_at = ?,           -- 当前时间戳
    retry_count = retry_count + 1
WHERE id = (
  SELECT id FROM tasks
  WHERE status = 'pending'
    AND scheduledAt IS NOT NULL
    AND scheduledAt <= ?     -- 到达排期时间
  ORDER BY scheduledAt ASC
  LIMIT 1
)
RETURNING *;
```

这是一个**原子的 SELECT-FOR-UPDATE 模式**（SQLite 单写者模型天然保证）。

#### 4.1.3 重试机制

**位置**: `queueService.ts:188-207` (`failTask`)

```sql
UPDATE tasks
SET errorMsg = ?,
    status = CASE WHEN retry_count >= 3 THEN 'failed' ELSE 'pending' END,
    locked_at = NULL
WHERE id = ?
```

- 最多重试 3 次
- 每次 `acquireNextTask` 时 `retry_count += 1`
- 达到 3 次后永久标记为 `failed`

#### 4.1.4 僵尸任务恢复

**位置**: `queueService.ts:61-89` (`markStalledTasksAsFailed`)

**启动时调用** (index.ts:299):
```typescript
QueueService.getInstance().markStalledTasksAsFailed()
```

默认超时: 300,000ms (5分钟)

策略：所有 `status='processing'` 且 `locked_at < 当前时间-5分钟` 的任务 → 标记为 `failed`，写入错误信息 `[System] 异常中断：发布期间应用退出或崩溃`。

⚠️ **注意**: 当前只在启动时调用 `markStalledTasksAsFailed`，不调用 `recoverStalledTasks`。也就是说，**崩溃后的任务不会自动重试，而是直接标记失败**。`recoverStalledTasks` 方法虽然存在但未被使用。

#### 4.1.5 并发控制

系统有**三层**防并发：

1. **isPublishing 内存锁** (publisher.ts:209, 437): `PublisherService` 实例级别的布尔锁
2. **runningByAccount Map** (publisher.ts:570): IPC handler 级别，防止同一账号重复触发
3. **runningScheduleByAccount Map** (index.ts:57): 心跳调度级别，防止同一账号被多次调度

⚠️ 但这些都是**内存锁**，不是数据库锁。如果进程崩溃，锁就丢失了（这就是 `markStalledTasksAsFailed` 存在的原因）。

### 4.2 资产管理 (Asset System)

#### 4.2.1 safe-file:// 协议

**位置**: `index.ts:363-643`

`safe-file://` 是一个自定义的 Electron protocol，作用：
- 渲染进程不能直接通过 `file://` 访问本地文件（安全限制）
- `safe-file://` 替代 `file://`，由主进程统一处理
- 支持 HTTP Range 请求（视频播放需要 seek）

#### 4.2.2 智能回退 (Smart Fallback) — 核心防御逻辑

**位置**: `index.ts:424-563`

当 `safe-file://` 请求的原始路径不存在时，系统会执行一个**多层回退搜索**：

```
第 1 层: macOS 本地化路径修复 (行 405-422)
  /Users/z/桌面/  →  /Users/z/Desktop/
  /Users/z/文稿/  →  /Users/z/Documents/
  等 6 对中英文映射

第 2 层: 内存缓存查找 (行 442-447)
  safeFileRecoveredByName Map 缓存上次成功的路径

第 3 层: 固定目录精确匹配 (行 448-477)
  按文件名在以下目录查找:
  - {workspace}/assets/images/{filename}
  - {workspace}/assets/{filename}
  - {userData}/generated_assets/{filename}
  同时尝试大小写变体 (.jpg / .JPG)

第 4 层: 大小写不敏感遍历 (行 479-550)
  对上述 3 个目录做 readdir + 逐文件比较 toLowerCase

第 5 层: 递归搜索 (行 500-561)
  BFS 搜索 {workspace}/assets/ 目录树
  最大深度: 4 层
  最大访问目录数: 6000
```

**为什么需要这么复杂？**
- 用户可能在 macOS 和 Windows 之间迁移工作区
- HEIC 转 JPG 后文件名变成 hash 值
- 旧 JSON 数据中存的是绝对路径，换电脑后路径全部失效
- macOS 中文化的文件夹名 (`桌面` vs `Desktop`)

**缓存机制**: 一旦通过回退找到文件，结果会缓存到 `safeFileRecoveredByName` Map 中，后续请求直接命中。

#### 4.2.3 资产入库 (Asset Localization)

**位置**: `taskManager.ts:309-589` (`createBatch`)

创建任务时的图片处理流程：

```
原始图片路径
    │
    ├─ HTTP URL → 直接保存 URL，不下载
    │
    ├─ 以 assets/ 开头 → 视为已入库的相对路径，直接使用
    │
    ├─ 在 generated_assets 目录下 → 直接使用绝对路径
    │
    └─ 外部绝对路径 →
        ├─ .heic 文件:
        │   1. 读取原文件 → heic-convert → JPEG Buffer
        │   2. 计算 SHA1 hash
        │   3. 写入 {workspace}/assets/images/{hash}.jpg
        │
        └─ 其他图片:
            1. 计算文件 SHA1 hash
            2. copyFileWithRetry → {workspace}/assets/images/{hash}{ext}
               (3次重试, 指数退避, 20秒超时)
            3. 如果策略是 'move'，删除源文件
```

**去重**: 通过 SHA1 hash 命名。相同内容的文件只会存一份。

#### 4.2.4 Remix 模式的图片变异

当任务的 `tags` 包含 `'裂变'` 或 `'remix'` 时：

**图片变异** (`imageMutator.ts`):
- 1% 裁切边距
- 亮度抖动 ±1%
- 重新编码 (quality=92)
- 输出到 `{userData}/generated_assets/mutated_{uuid}.jpg`

**文本变异** (`textSpinner.ts`):
- 24 组中文同义词 (小红书风格)
- 30%-50% 概率替换
- 最长匹配优先

### 4.3 账号管理

**当前状态**: 存在两套 AccountManager！

| 文件 | 存储 | 使用状态 |
|------|------|----------|
| `src/main/accountManager.ts` | ElectronStore (JSON) | ✅ 主进程实际使用 |
| `src/main/services/accountManager.ts` | SQLite | ❌ 未接入主流程 |

**实际使用的是 JSON 版** (index.ts:320): `new AccountManager(workspaceDbStore)`

每个账号有独立的 `partitionKey`，格式为 `persist:xhs_${uuid}`。Electron 的 `partition` 机制会为每个 key 维护独立的 Cookie/Session 存储。

### 4.4 飞书集成

**位置**: `index.ts:120-156, 1555-1706`

功能：将任务数据（含图片）上传到飞书多维表格。

流程：
1. 获取 `tenant_access_token`（有 60 秒安全裕量的缓存）
2. 上传图片到飞书 Drive API
3. 创建多维表格记录

⚠️ **字段兼容 hack** (`normalizeBitableAttachmentFields`, 行 158-189): 飞书 API 返回的附件字段有时用 `image_key` 而不是 `file_token`，此函数做了自动转换。

### 4.5 视频处理

**位置**: `services/videoProcessor.ts`

- 使用 `ffprobe` 检测视频编解码器
- 兼容编码: H.264, VP8, VP9
- 不兼容时用 `ffmpeg` 转码为 H.264/AAC MP4
- 预览文件缓存在 `{userData}/temp_previews/`
- 启动时清理超过 60 个或 7 天的预览文件

### 4.6 图片处理工具箱

| 功能 | IPC Channel | 外部依赖 |
|------|------------|----------|
| 九宫格切图 | `process-grid-split` | sharp |
| HD 超分辨率 | `process-hd-upscale` | Real-ESRGAN 可执行文件 |
| 水印去除 | `process-watermark` | Python cms_engine (IOPaint/LAMA) |

**GPU 限流**: `gpuLimit = pLimit(1)`，所有 GPU 密集操作串行执行。

**打包策略**:
- 开发环境: 用户配置 Python 路径和脚本路径
- 生产环境: 使用 PyInstaller 打包的 `cms_engine` 二进制 (`resources/cms_engine`)

**位置**: `taskManager.ts:46` 和 `index.ts:787`

```typescript

```

旧版本使用 `'success'` 作为完成状态，新版本是 `'published'`。此映射保证旧数据兼容。

### 5.5 scheduledAt 三重字段兼容

**位置**: `sqliteService.ts:352-356`

```typescript
const rawTime = t.scheduledAt ?? t.scheduleTime ?? t.startTime
```

三个不同历史版本使用了不同的字段名存储排期时间。

### 5.6 CRITICAL 日志守卫

**位置**: `queueService.ts:163-171`

```typescript
if (task && task.scheduledAt === null) {
  console.error('CRITICAL: Queue picked up an unscheduled task!', task.id)
  // 回滚状态
  sqlite.connection.prepare(`UPDATE tasks SET status = 'pending', locked_at = NULL WHERE id = ?`).run(task.id)
  return null
}
```

这是一个**不应该发生的守卫**。如果 SQL WHERE 条件正确，`scheduledAt` 不可能为 null。但这个检查防止了极端情况下未排期任务被错误执行。

## 六、数据流总览

```
渲染进程 (React + Zustand)
    │
    │  IPC (contextBridge)
    ▼
主进程 (index.ts)
    │
    ├── AccountManager (JSON) ──── ElectronStore → db.json
    ├── ProductManager (JSON) ──── ElectronStore → db.json
    ├── TaskManager ──────────── SqliteService → cms.sqlite
    ├── QueueService ─────────── SqliteService → cms.sqlite
    ├── PublisherService ─────── 创建 Worker Window
    │       │
    │       └── xhs-automation.ts (preload) ── DOM 自动化
    │
    ├── WorkspaceService ─────── configStore (ElectronStore)
    ├── VideoProcessor ──────── ffmpeg/ffprobe
    └── ImageLab ────────────── sharp / Real-ESRGAN / cms_engine
```

⚠️ **双存储并存**: 账号和商品仍在 JSON，任务已迁移到 SQLite。这是半完成的迁移状态。
